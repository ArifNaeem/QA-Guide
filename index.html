1. Software Development Life Cycle (SDLC)

The SDLC is a structured process used by software development teams to design, develop, test, and deploy high-quality software. It provides a framework defining tasks performed at each step.

Phases of SDLC

  1. Requirement Gathering & Analysis: Business analysts collect requirements from stakeholders and document them in a BRD (Business Requirement Document) or SRS (Software Requirement Specification).
  2. Planning / Feasibility Study: Evaluates technical, operational, and economic feasibility of the project.
  3. Design: High-Level Design (HLD) describes architecture; Low-Level Design (LLD) details modules/components.
  4. Implementation / Coding: Developers write code per the design documents.
  5. Testing: QA team validates the software against requirements and reports defects.
  6. Deployment: Release to production environment for end-user use.
  7. Maintenance: Ongoing bug fixes, enhancements, and updates.

2. Software Testing Life Cycle (STLC)

STLC is a sequence of specific activities conducted during the testing process to ensure software quality goals are met.

Phases of STLC

  1. Requirement Analysis: Testers study requirements to identify testable items. Deliverable: RTM (Requirement Traceability Matrix).
  2. Test Planning: Test lead prepares the Test Plan — defines scope, approach, resources, schedule. Deliverable: Test Plan, Effort Estimation.
  3. Test Case Design: Writing detailed test cases and preparing test data. Deliverable: Test Cases, Test Scripts.
  4. Test Environment Setup: Configuring hardware/software needed for testing. Deliverable: Environment Readiness Report.
  5. Test Execution: Running test cases, reporting defects, and retesting. Deliverable: Test Execution Report, Defect Report.
  6. Test Closure: Analysis of results, lessons learned, metrics. Deliverable: Test Closure Report.

SDLC vs STLC

Covers entire software developmentCovers only the testing portion
Focuses on building the productFocuses on validating the product
Involves BA, developers, QA, stakeholdersInvolves QA team only
Deliverable: Working softwareDeliverable: Quality report, defect report

3. Types of Testing

A. Functional Testing

Verifies that each function of the software operates per requirements. Input-output based.

  • Unit Testing: Individual components/modules (done by developers).
  • Integration Testing: Modules combined and tested together.
  • System Testing: Complete, integrated system tested end-to-end.
  • Acceptance Testing: Performed by client/end user (UAT).

B. Non-Functional Testing

  • Performance Testing: How system performs under normal load.
  • Load Testing: Behavior under expected load conditions.
  • Stress Testing: Behavior beyond normal capacity (break point).
  • Volume Testing: Handling large amounts of data.
  • Security Testing: Identifies vulnerabilities.
  • Usability Testing: User-friendliness of interface.
  • Compatibility Testing: Across browsers, OS, devices.
  • Recovery Testing: System's ability to recover from crashes.

C. Other Important Types

  • Smoke Testing: Also called "Build Verification Testing." Quick check to ensure critical functionality works after a new build.
  • Sanity Testing: Narrow regression after minor changes to verify specific fixes work.
  • Regression Testing: Re-running tests to ensure new changes don't break existing functionality.
  • Retesting: Re-executing the same test case that failed to verify the fix.
  • Exploratory Testing: Simultaneous learning, test design, and execution without pre-written cases.
  • Ad-hoc Testing: Informal testing without documentation — "random testing."
  • Monkey Testing: Random inputs to check system stability.
  • Alpha Testing: Done in-house by QA team before release.
  • Beta Testing: Done by real users in real environment before production launch.

Smoke vs Sanity Testing

Smoke TestingSanity Testing
Broad but shallowNarrow but deep
Performed on initial buildPerformed on stable build after minor changes

4. Test Case Design

Components of a Good Test Case

FieldDescription
Test Case IDUnique identifier (e.g., TC_Login_001)
Test Scenario / TitleBrief description of what is being tested
Description / ObjectivePurpose of the test case
PreconditionsSetup required before execution
Test StepsStep-by-step instructions
Test DataInputs required
Expected ResultWhat should happen
Actual ResultWhat actually happened
StatusPass / Fail / Blocked / Skipped
PriorityHigh / Medium / Low
SeverityCritical / Major / Minor / Trivial
Comments / NotesAdditional info, screenshots

Test Case Design Techniques

  • Equivalence Partitioning (EP): Divide input data into valid and invalid partitions. Test one value from each partition.
    Example: Age field (1–100). Valid: {1–100}, Invalid: {<1, >100}. Test with 50, 0, and 101.
  • Boundary Value Analysis (BVA): Test at boundaries where defects cluster. For range 1–100, test 0, 1, 2, 99, 100, 101.
  • Decision Table Testing: Tabular representation of inputs vs expected outputs — used for complex business rules.
  • State Transition Testing: Used when behavior depends on state (e.g., ATM: Card Inserted → PIN Entered → Authenticated).
  • Use Case Testing: Based on actor-system interactions.
  • Error Guessing: Testers use experience to predict where errors might occur.
  • Pair-wise / Orthogonal Array Testing: Covers all pairs of parameter values efficiently.
Tip: Write test cases that are atomic (one check per case), reusable, independent, and traceable to requirements.

5. Defect Life Cycle (Bug Life Cycle)

The defect life cycle represents the various states a bug passes through from discovery to closure.

Defect States

  1. New: Defect is logged for the first time.
  2. Assigned: Assigned to a developer by the lead/manager.
  3. Open: Developer starts analyzing the defect.
  4. Fixed: Developer has fixed the defect.
  5. Pending Retest / Ready for Test: Moved to QA for retesting.
  6. Retest: Tester verifies the fix.
  7. Verified: Fix works correctly.
  8. Closed: Defect fully resolved.
  9. Reopened: Fix didn't work — sent back to developer.
  10. Rejected: Not a valid defect.
  11. Deferred: Valid but postponed to future release.
  12. Duplicate: Already reported.

Severity vs Priority

SeverityPriority
Impact of defect on functionalityHow urgently it needs to be fixed
Set by testerSet by manager/product owner
Levels: Critical, Major, Minor, TrivialLevels: High, Medium, Low
Technical aspectBusiness aspect

Examples

  • High Severity, High Priority: Login page crashes.
  • High Severity, Low Priority: A rarely-used page fails under extreme load.
  • Low Severity, High Priority: Company logo misspelled on homepage.
  • Low Severity, Low Priority: Minor alignment issue on settings page.

6. Bug Reporting Tools

  • JIRA (Atlassian): Most popular tool for Agile teams — issue tracking, sprint management, dashboards.
  • Bugzilla: Open-source web-based bug tracking system.
  • TestRail: Test case management with bug tracker integration.
  • Mantis BT: Open-source bug tracker.
  • Redmine: Project management + issue tracking.
  • Azure DevOps: Microsoft's integrated DevOps platform.
  • Zephyr: Test management for JIRA.
  • Trello / Asana: Simple task & bug tracking for smaller teams.

Elements of a Good Bug Report

  1. Bug ID & Title (clear & descriptive)
  2. Environment details (OS, browser, version, device)
  3. Build / Version
  4. Steps to reproduce (numbered, clear)
  5. Expected result
  6. Actual result
  7. Severity & Priority
  8. Screenshots / videos / logs
  9. Reproducibility (always, sometimes, once)
  10. Assignee

7. Test Plan and Test Strategy

Test Plan

A document describing the scope, approach, resources, and schedule of testing activities. Created by the Test Manager.

Components (per IEEE 829):
  1. Test Plan Identifier
  2. Introduction
  3. Test Items
  4. Features to be Tested
  5. Features NOT to be Tested
  6. Approach
  7. Pass/Fail Criteria
  8. Suspension & Resumption Criteria
  9. Test Deliverables
  10. Environmental Needs
  11. Staffing & Training Needs
  12. Responsibilities
  13. Schedule
  14. Risks & Contingencies
  15. Approvals

Test Strategy

A high-level document outlining the overall testing approach at the organizational level. Usually static and project-independent.

8. Test Data Management

Test data management involves creating, storing, and maintaining data needed for test execution.

Types of Test Data

  • Valid Data: Correctly formatted, within allowed range.
  • Invalid Data: Wrong format or out of range — tests error handling.
  • Boundary Data: At edges of valid ranges.
  • Null / Empty Data: Testing blank field handling.
  • Stress / Huge Data: For performance/volume testing.

Best Practices

  • Mask / anonymize production data to respect privacy (GDPR).
  • Use test data generation tools (Mockaroo, Faker).
  • Maintain reusable test data sets.
  • Keep data isolated between environments.

9. Testing Techniques

Black Box Testing

Tester examines functionality without knowing internal code structure. Focus is on inputs and expected outputs.

  • Examples: Equivalence partitioning, BVA, decision tables.
  • Used for functional, system, acceptance testing.

White Box Testing

Tester has full knowledge of code/internal logic. Focus is on code paths, branches, loops.

  • Examples: Statement coverage, branch coverage, path coverage.
  • Used for unit testing.

Gray Box Testing

Combination — tester has partial knowledge of internal structure. Commonly used for integration and security testing.

10. Cross-Browser and Cross-Device Testing

Ensures the application behaves consistently across various browsers, OS, and devices.

Key Areas

  • Browsers: Chrome, Firefox, Safari, Edge, Opera.
  • Operating Systems: Windows, macOS, Linux, iOS, Android.
  • Devices: Desktop, tablets, mobile (varied screen sizes/resolutions).
  • Responsive design: Layout adapts to screen size.

Tools

  • BrowserStack, Sauce Labs, LambdaTest — cloud-based testing.
  • Chrome DevTools — device emulation.
  • Responsinator — quick responsive checks.

11. Security Testing

Identifies vulnerabilities to protect data and functionality from threats.

OWASP Top 10 Vulnerabilities (Must Know)

  1. Broken Access Control
  2. Cryptographic Failures
  3. Injection (SQL, NoSQL, OS Command)
  4. Insecure Design
  5. Security Misconfiguration
  6. Vulnerable & Outdated Components
  7. Identification & Authentication Failures
  8. Software & Data Integrity Failures
  9. Security Logging & Monitoring Failures
  10. Server-Side Request Forgery (SSRF)

Common Attacks to Test

  • SQL Injection: ' OR '1'='1 in login fields.
  • XSS (Cross-Site Scripting): <script>alert('xss')</script>.
  • CSRF: Forged request from authenticated user.
  • Session Hijacking: Stealing session tokens/cookies.
  • Brute Force: Guessing credentials by repeated attempts.

Security Testing Tools

  • OWASP ZAP, Burp Suite, Nessus, Nikto, Acunetix.

12. Usability Testing

Evaluates how user-friendly, intuitive, and accessible the application is for end users.

Key Parameters (The 5 E's)

  • Effective: Does it achieve user goals?
  • Efficient: Can users complete tasks quickly?
  • Engaging: Is it pleasant to use?
  • Error-tolerant: Does it prevent/recover from errors?
  • Easy to learn: Can a first-time user pick it up?

Areas to Check

  • Navigation clarity
  • Consistency in design (colors, fonts, buttons)
  • Readability of text
  • Accessibility (WCAG — color contrast, screen reader support, keyboard navigation)
  • Meaningful error messages
  • Mobile-friendliness

13. Test Execution and Reporting

Execution Steps

  1. Pick test cases from the repository.
  2. Prepare test environment and data.
  3. Execute each step and compare actual with expected.
  4. Mark status: Pass / Fail / Blocked / Skipped.
  5. Log defects for failed cases.
  6. Attach evidence (screenshots, logs).

Test Metrics to Report

  • Total test cases executed vs. planned
  • Pass/Fail percentage
  • Defect density (defects per module/KLOC)
  • Defect severity index
  • Test execution progress
  • Defect removal efficiency (DRE)

Daily Status Report (DSR) Contents

  • Date, tester name, module
  • Planned vs. executed cases
  • Pass/fail counts
  • Blockers / risks
  • Defects logged

14. Test Environment Setup

The test environment is a setup of hardware, software, and network configurations where testing takes place.

Elements

  • Hardware (servers, client machines)
  • Software (OS, DB, browsers, dependencies)
  • Network configuration
  • Test data setup
  • Test tools (Selenium, JIRA, Postman etc.)
  • Build artifacts deployment

Common Environments

  • DEV: Developer's local environment
  • QA / Test: For QA team
  • Staging / Pre-Prod: Mirror of production
  • UAT: For end-user validation
  • Production: Live environment

15. Risk-Based Testing (RBT)

An approach that prioritizes tests based on the risk (likelihood × impact) of failure.

Steps

  1. Identify risks (product & project).
  2. Assess risk probability & impact.
  3. Prioritize risks (High / Medium / Low).
  4. Design & execute tests based on priority.
  5. Monitor & mitigate risks throughout project.

Risk Matrix

Impact →LowMediumHigh
High ProbabilityMediumHighCritical
Medium ProbabilityLowMediumHigh
Low ProbabilityLowLowMedium

16. Agile Methodology in Testing

Agile Principles (Relevant)

  • Working software over comprehensive documentation
  • Customer collaboration over contract negotiation
  • Responding to change over following a plan
  • Continuous delivery of value

Scrum Framework

  • Roles: Product Owner, Scrum Master, Dev Team.
  • Artifacts: Product Backlog, Sprint Backlog, Increment.
  • Ceremonies:
    • Sprint Planning
    • Daily Stand-up (15 mins)
    • Sprint Review
    • Sprint Retrospective
    • Backlog Grooming / Refinement
  • Sprint: 1–4 weeks iteration.

Kanban

Visual workflow (To Do → In Progress → Testing → Done). Focus on limiting WIP and continuous flow.

Scrum vs Kanban

ScrumKanban
Time-boxed sprintsContinuous flow
Fixed rolesNo prescribed roles
Sprint-based planningOn-demand planning

Agile Testing Practices

  • TDD (Test-Driven Development): Write test first → write code to pass test → refactor.
  • BDD (Behavior-Driven Development): Gherkin syntax — Given/When/Then.
  • ATDD (Acceptance Test-Driven Development): Acceptance criteria drive tests.
  • CI/CD: Continuous Integration & Deployment — automated build and test pipelines.
  • Shift-Left Testing: Start testing earlier in SDLC.

Agile QA Responsibilities

  • Participate in all ceremonies
  • Help define acceptance criteria
  • Write automated & manual tests per user story
  • Perform exploratory testing
  • Collaborate with devs continuously

17. Test Automation Fundamentals

Why Automate?

  • Speed & efficiency for regression
  • Repeatability & reliability
  • Cost saving in the long run
  • Coverage for data-driven scenarios

What to Automate

  • Regression tests, smoke tests
  • Data-driven tests
  • Repetitive tests
  • Performance & load tests

What NOT to Automate

  • Exploratory testing
  • Usability testing
  • Tests that run rarely
  • Constantly changing UI

18. Soft Skills for QA Engineers

  • Communication: Clearly articulate bugs, risks, and requirements to developers, BAs, POs.
  • Analytical thinking: Break down complex problems, think about edge cases.
  • Attention to detail: Notice subtle UI inconsistencies, off-by-one errors, typos.
  • Curiosity: Ask "what if?" — explore beyond happy paths.
  • Time management: Prioritize testing with limited time.
  • Collaboration: Work closely with devs, designers, product owners.
  • Adaptability: Requirements change, priorities shift — remain flexible.
  • Critical thinking: Don't assume — question, verify.
  • Empathy: Think like end users, feel their pain points.
  • Patience & persistence: Reproducing hard-to-find bugs takes time.

19. Ethical Hacking (Basics for Security Testing)

Ethical hackers identify vulnerabilities legally to help strengthen security.

Phases of Ethical Hacking

  1. Reconnaissance: Gathering information (passive & active).
  2. Scanning: Port scanning, vulnerability scanning.
  3. Gaining Access: Exploiting vulnerabilities.
  4. Maintaining Access: Persistence (for legitimate long-term testing).
  5. Clearing Tracks / Reporting: Document findings.

Types of Hackers

  • White Hat: Ethical hackers (authorized).
  • Black Hat: Malicious attackers.
  • Gray Hat: In between — unauthorized but not malicious.

Basic Tools for QA

  • Burp Suite — web proxy & testing.
  • OWASP ZAP — automated scanner.
  • Nmap — network discovery.
  • Wireshark — network packet analysis.
  • Metasploit — exploitation framework.
Note: Always get written authorization before performing any security testing.

20. Industry Standards and Compliance

Testing Standards

  • ISO/IEC 29119: International standard for software testing (processes, documentation, techniques).
  • ISO 9126 / 25010: Software quality model (functionality, reliability, usability, efficiency, maintainability, portability).
  • IEEE 829: Standard for test documentation (test plan template).
  • IEEE 1008: Standard for unit testing.
  • CMMI: Capability Maturity Model Integration (Levels 1–5).

Compliance Regulations

  • GDPR (EU): Data protection & privacy — test for data masking, consent, right-to-erasure.
  • HIPAA (US): Healthcare data security & privacy.
  • PCI-DSS: Payment card industry standards.
  • SOX: Financial reporting accuracy.
  • WCAG: Web content accessibility.
  • FDA 21 CFR Part 11: Electronic records in pharma / medical devices.

21. Common Interview Questions & Sample Answers

Q1. What is the difference between verification and validation?
Verification: "Are we building the product right?" — static testing, reviews, inspections.
Validation: "Are we building the right product?" — dynamic testing, actual execution.
Q2. What is the difference between a bug, defect, error, and failure?
Error: Mistake made by developer in code.
Bug/Defect: Result of the error found during testing.
Failure: When the defect reaches the end user and causes malfunction.
Q3. When do you stop testing?
When: test cases are executed with acceptable pass %, test coverage & exit criteria are met, deadline is reached, no critical/high defects are open, and the product is stable.
Q4. What is a Traceability Matrix?
A document that maps requirements to test cases to ensure all requirements are tested — also called RTM (Requirement Traceability Matrix).
Q5. Difference between positive and negative testing?
Positive: Verify the system works with valid inputs.
Negative: Verify the system handles invalid inputs gracefully.
Q6. What is a test harness?
A collection of software & test data used to test a program by running it under varying conditions and monitoring its behavior and outputs.
Q7. What is a test bed?
An environment configured with hardware, software, network, and test data used to execute tests.
Q8. What are test deliverables?
Documents produced during testing: test plan, test cases, test scripts, test data, RTM, defect reports, test summary report.
Q9. What is Boundary Value Analysis? Give an example.
Testing at the boundaries of input ranges where defects cluster. For a field that accepts 1–100, test values: 0, 1, 2, 99, 100, 101.
Q10. Explain the bug life cycle.
New → Assigned → Open → Fixed → Retest → Verified → Closed. May also include: Reopened, Rejected, Deferred, Duplicate.
Q11. What's the difference between Alpha and Beta testing?
Alpha: Done in-house by testers/developers before release.
Beta: Done by real users in their environment before final release.
Q12. What is the difference between test scenario and test case?
Test Scenario: High-level "what to test" (e.g., verify login functionality).
Test Case: Detailed step-by-step "how to test" with inputs & expected results.
Q13. What are exit criteria?
Conditions that must be met before testing is considered complete — e.g., 100% test execution, 95% pass rate, no critical open defects.
Q14. What is the difference between load testing and stress testing?
Load: System behavior under expected load.
Stress: System behavior beyond capacity to find breaking point.
Q15. What is API testing, and how is it different from UI testing?
API testing verifies backend services (request/response, status codes, data integrity) — faster, reliable, no UI. UI testing verifies the front-end — slower, brittle, requires rendering.
Q16. What is the difference between test plan and test strategy?
Test plan is project-specific and dynamic, derived from SRS, created by test manager. Test strategy is organization-wide and static, derived from business requirements, created by PM.
Q17. How do you prioritize test cases?
Based on: business criticality, risk of failure, frequency of use, impact on end users, requirements priority, and complexity of the module.
Q18. How do you handle a situation where a developer says "it's not a bug"?
Stay professional. Provide clear reproduction steps, evidence (screenshots/logs), reference to requirements or user expectations. Involve BA/PO if disagreement persists.
Q19. What is a defect leakage?
When a defect escapes from a testing phase to production or a later phase. Metric: Defect Leakage = (Defects in prod / Total defects) × 100.
Q20. What's your approach when you have limited time to test?
Use risk-based testing: focus on critical functionalities, high-risk areas, recently changed code. Perform exploratory & smoke tests first. Communicate risks to stakeholders.
